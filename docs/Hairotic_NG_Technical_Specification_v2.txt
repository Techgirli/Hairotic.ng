1. Purpose & Scope
This document is the implementation blueprint for Hairotic.ng: it defines the technology stack, system architecture, data flows, database schema, API surface, folder structure, deployment strategy, and phased implementation plan. It complements the Comprehensive PRD (which defines what to build and why) by defining exactly how to build it.
Every technology choice below is deliberately free-tier or near-free at MVP scale — AWS is explicitly excluded per project requirements. Each choice documents a clear upgrade path so growth never forces a re-architecture, only a billing tier change.
2. Technology Stack
The stack favors managed, serverless-friendly services with generous free tiers, type safety end-to-end (TypeScript + Zod + Prisma), and a small number of moving parts appropriate for a lean engineering team.

3. System Architecture
The architecture is a straightforward layered web application: an edge/CDN layer, a server-rendered frontend, a REST API, a relational datastore with a cache, media storage, and a set of external services for payments, email, and WhatsApp messaging. No component requires AWS.

4. Module Boundaries (Composable Commerce)
Matching the PRD's composable architecture, the NestJS backend is organized as one module per business capability. At MVP scale all modules run inside a single deployable service with clear internal boundaries — this keeps operations simple while preserving the option to extract any module into its own service later without a rewrite.
auth/ — registration, login, refresh tokens, MFA, RBAC guards
catalog/ — products, variants, categories, collections, search
cart/ — guest (Redis) and account (Postgres) cart state
checkout/ — order creation, stock holds, idempotency handling
payments/ — Paystack init, webhook verification, refunds
orders/ — order lifecycle, status history, guest tracking lookup
reviews/ — verified-purchase reviews and photo moderation
wishlist/ — saved products per customer
notifications/ — Resend transactional email
whatsapp/ — Meta WhatsApp Cloud API integration
analytics/ — event ingestion and funnel/report queries
admin/ — admin-only endpoints layered with stricter RBAC + audit logging
5. High-Level Data Flows
The most business-critical flow — checkout and payment confirmation — is diagrammed below, followed by a written breakdown of every major flow in the system.

Browse / Catalog Read Flow
Browser requests a page (Landing, Shop, Collection, or Product Page).
Next.js (Vercel) renders via SSR/ISR, calling GET endpoints on the NestJS API for product/collection data not already cached at the edge.
NestJS API queries PostgreSQL (via Prisma) for product, variant, and stock data.
Response is cached at the ISR layer (e.g., 5-minute revalidation for bestsellers) and served to the browser.
Authentication Flow
Browser submits email/password (or OAuth, if added later) to POST /auth/login.
NestJS API verifies the bcrypt password hash against the stored user record in PostgreSQL.
On success, API issues a short-lived JWT access token and a longer-lived refresh token, both set as httpOnly, Secure, SameSite=Strict cookies.
Subsequent requests include the access token; when it expires, the frontend silently calls /auth/refresh using the refresh token to obtain a new access token.
Cart & Checkout Flow
Guest or logged-in customer adds items to cart; cart state is held in Upstash Redis (guest, keyed by session) or PostgreSQL (logged-in, keyed by customer_id).
At Checkout submission, the API re-validates stock and price server-side, then creates an Order record in `pending_payment` status inside a database transaction, placing a short-TTL stock hold in Redis.
The API returns a Paystack transaction initialization response; the browser is redirected to Paystack's hosted checkout.
Payment Confirmation Flow
Customer completes payment on Paystack (card, bank transfer, or USSD).
Paystack sends a signed webhook to a dedicated NestJS endpoint (POST /payments/webhook).
The API verifies the webhook signature against the Paystack secret, then calls Paystack's verify-transaction API as a second confirmation before trusting the payload.
On confirmed success, the API updates the Order to `paid` inside a DB transaction with row-level locking on the relevant stock rows (preventing overselling), and clears the Redis stock hold.
The Notifications module triggers a Resend confirmation email and a WhatsApp Cloud API confirmation message in parallel.
A scheduled reconciliation job (GitHub Actions cron or Render cron job) re-verifies any order stuck in `pending_payment` for more than 10 minutes as a webhook-failure fallback.
Order Fulfillment & Tracking Flow
Admin updates order status (Processing → Shipped → Delivered) via the Admin Orders screen.
Each transition is written to `order_status_history` and triggers a WhatsApp/email update to the customer.
The customer-facing Order Tracking screen reads the same `order_status_history` table to render a live timeline, for both logged-in customers and guest lookups (order number + email/phone).
Media Upload Flow (Admin)
Admin uploads product/review images via the Admin Product Management screen.
The frontend requests a signed upload URL/preset from the NestJS API.
The browser uploads directly to Cloudinary using the signed credentials (bypassing the API for the binary payload).
Cloudinary returns the resulting URL, which the frontend sends to the API to persist against the relevant `product_images` or `review_photos` record in PostgreSQL.
6. Database Schema
PostgreSQL (hosted on Neon), accessed via Prisma. Core entities and their key fields:

--- Table Start ---
Table | Key Fields

addresses | id, user_id, label, state, lga, street, phone, is_default

product_variants | id, product_id, sku, attributes (length/color/texture), price, compare_at_price

inventory | id, product_variant_id, quantity, low_stock_threshold, updated_at
carts / cart_items | id, customer_id or session_id, product_variant_id, quantity
orders | id, customer_id (nullable for guest), status, subtotal, delivery_fee, total, idempotency_key, created_at
order_items | id, order_id, product_variant_id, quantity, unit_price
order_status_history | id, order_id, status, note, changed_by, created_at
payments | id, order_id, paystack_reference, status, amount, verified_at
reviews | id, product_id, customer_id, order_id, rating, body, verified_purchase
review_photos | id, review_id, url
wishlists | id, customer_id, product_variant_id, created_at
customer_notes | id, customer_id, admin_id, note, created_at
audit_logs | id, actor_id, action, entity, entity_id, before, after, created_at
--- Table End ---

Schema Notes
All monetary values stored as integers in kobo (smallest currency unit) to avoid floating-point rounding errors.
order_status_history is append-only and is the single source of truth for both the Admin Orders screen and the customer-facing Order Tracking screen.
audit_logs captures every admin/security-relevant mutation (actor, action, entity, before/after) per the Security Suite's audit requirements.
Stock decrement on payment confirmation uses a transaction with `SELECT ... FOR UPDATE` row locking on the relevant product_variants/inventory row to guarantee no overselling under concurrent checkouts.
7. API Design
REST API served by NestJS, versioned under /api/v1. All admin routes additionally require RBAC role checks beyond authentication.

--- Table Start ---
Route Group | Key Endpoints
/auth | POST /register, POST /login, POST /refresh, POST /logout, POST /mfa/verify
/users (customer profile) | GET /me, PATCH /me, GET /me/addresses, POST /me/addresses
/products | GET /products, GET /products/:slug, GET /collections, GET /collections/:slug
/cart | GET /cart, POST /cart/items, PATCH /cart/items/:id, DELETE /cart/items/:id
/checkout | POST /checkout (creates pending order + Paystack init)
/payments | POST /payments/webhook, GET /payments/:reference/verify
/orders | GET /orders (customer's own), GET /orders/:id, GET /orders/track (guest lookup)
/reviews | POST /products/:id/reviews, GET /products/:id/reviews
/wishlist | GET /wishlist, POST /wishlist, DELETE /wishlist/:id
/contact | POST /contact
/admin/products | Full CRUD + bulk actions, restricted to Admin/Catalog-Manager role
/admin/inventory | GET /admin/inventory, PATCH /admin/inventory/:id (with reason code)
/admin/orders | GET /admin/orders, PATCH /admin/orders/:id/status, POST /admin/orders/:id/refund
/admin/customers | GET /admin/customers, GET /admin/customers/:id, POST /admin/customers/:id/notes
/admin/analytics | GET /admin/analytics/summary, GET /admin/analytics/funnel, GET /admin/analytics/export
--- Table End ---

Conventions
All list endpoints support cursor-based pagination via `?cursor=` and `?limit=` query params.
All write endpoints validate request bodies against Zod (frontend) and class-validator (backend) schemas derived from the same shape where practical.
All mutating endpoints that create financial records (checkout, refunds) require an `Idempotency-Key` header to prevent duplicate processing on retry.
Errors follow a consistent shape: `{ statusCode, message, error, requestId }` to simplify frontend handling and support tracing via Sentry.

CI/CD Pipeline (GitHub Actions)
On every pull request: install dependencies, run lint (ESLint), run type-check (tsc), run unit + integration tests, run a Prisma migration dry-run against a Neon preview branch.
On merge to main: run the same checks, then trigger Vercel production deploy and Render production deploy; run Prisma migrations against the production database as a controlled, reviewed step (never auto-applied without review for destructive changes).
Secrets (Paystack keys, JWT secret, database URL, Resend/WhatsApp tokens) are stored as encrypted GitHub Actions secrets and platform-native environment variables on Vercel/Render — never committed to source control.

Phased to deliver Must Have screens first, matching the PRD's priority ordering.

Set up Neon Postgres project (dev + staging branches) and Upstash Redis instance.
Configure Prisma schema from Section 6 (Database Schema) and run initial migration.

Build Auth module (register/login/refresh/MFA) and RBAC middleware.
Build Catalog module + Landing, Shop, Collections, Product Page screens.
Build Cart + Checkout modules and screens, with server-side price/stock re-validation.
Integrate Paystack (init + webhook + verify) and build the Payment screen/flow.

Phase 2 — Admin Back-Office (Must Have screens)
Build Admin Login (with mandatory MFA) and RBAC-scoped Admin Dashboard.

Phase 3 — Trust & Retention (Should Have screens)

Phase 4 — Growth (Could Have / Post-MVP)
Build Blog module and screen.

Full OWASP Top 10 review against the Security Suite checklist.
Load-test checkout/payment flow for race conditions (concurrent last-unit purchase).
Configure Sentry, UptimeRobot, and GitHub Actions deploy pipelines for staging + production.
Run a documented backup/restore drill against Neon's point-in-time recovery before go-live.
11. Security
Full implementation and operational security standards live in the dedicated Hairotic.ng Security Suite document. This spec implements the baseline controls it requires:
TLS/HTTPS everywhere, enforced at the Cloudflare edge and platform level (Vercel/Render both provide automatic HTTPS).
JWT access + refresh token rotation; bcrypt password hashing (cost factor ≥12).
RBAC guard applied at the NestJS controller level for every admin route, not just hidden in the UI.
Mandatory MFA for all admin/staff accounts.
Rate limiting via Upstash Redis on auth, checkout, contact-form, and review-submission endpoints.
Input validation on every mutating endpoint via class-validator (backend) mirroring Zod schemas (frontend).
Parameterized queries exclusively via Prisma — no raw string-concatenated SQL.
Paystack webhook signature verification on every payment webhook before trusting the payload.
Daily automated backups via Neon's built-in point-in-time recovery, with a documented quarterly restore drill.
Structured audit logging (audit_logs table) for all admin actions and security-relevant events.
12. Monitoring & Operations
Sentry captures unhandled exceptions on both frontend (Vercel) and backend (Render), with Slack/email alerting on new issue types.
UptimeRobot polls a `/health` endpoint on the API and the storefront homepage every 5 minutes, alerting on downtime, feeding the ≥99.9% uptime KPI.
Vercel Analytics provides Core Web Vitals and traffic data at no extra cost; combined with first-party event logging in Postgres for ecommerce funnel analysis (feeds the Admin Analytics screen).
A weekly manual or scheduled review of Sentry issues, UptimeRobot incidents, and low-stock alerts is recommended until traffic justifies a dedicated on-call rotation.
Disaster recovery: Neon point-in-time recovery covers database restore; Vercel/Render deployments are redeployable from Git history at any time, so full-stack recovery does not depend on manual server rebuilds.
