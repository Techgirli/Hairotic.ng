1. Executive Summary
Hairotic.ng is a premium Nigerian hair ecommerce brand built around trust, authenticity, and a bold, expressive identity (deep orange/white palette, confident brand voice). This document is the single source of truth for what to build, why, and how — detailed enough that a human engineering team or an AI coding agent can implement the platform consistently without ambiguity.
This edition treats the project like a flagship open-source-grade repository: every feature carries an objective, business value, user story, acceptance criteria, edge cases, dependencies, technical notes, analytics events, and security considerations. Every one of the 21 screens across the customer-facing storefront and admin back-office is documented to this standard.
A key constraint for this version: the platform must run entirely on free or near-free cloud tiers — AWS is explicitly excluded. Section 5 defines the replacement architecture.
2. Vision & Business Goals
Mission
To help every woman express her boldest self through hair that turns heads and never holds her back.
Vision
To become the number one name young, confident Nigerian women think of when they want hair that matches their energy — starting in Lagos, scaling nationwide.
Business Goals
Become a trusted, premium hair destination online.
Increase online sales volume and average order value.
Deliver an excellent, fast, mobile-first shopping experience.
Scale the platform securely without AWS-scale infrastructure cost.
Objectives & KPIs
Conversion rate > 2.5%
Platform uptime ≥ 99.9%
Page load time < 2.5 seconds on mobile
Repeat purchase rate > 20%
Net Promoter Score (NPS) > 50
3. Problem Statement
Customers shopping for hair online in Nigeria routinely struggle with counterfeit or misrepresented products, poor mobile UX, low trust in payment security, and fragmented, confusing checkout flows split across Instagram DMs, WhatsApp, and unreliable websites. Hairotic.ng solves this by combining a single trustworthy storefront with a familiar WhatsApp-based confirmation and support layer, transparent stock/pricing, and verified reviews.
4. Target Personas
Busy Professional (primary)
Female, 25-40, Lagos-based, values convenience and low-maintenance protective styles, willing to pay for guaranteed quality, shops mostly on mobile during commute/lunch breaks.
Beauty Entrepreneur
Runs her own salon or resells hair; buys in bulk, price-sensitive, needs fast WhatsApp communication and reliable stock availability.
First-Time Buyer
18-30, trend-driven, discovered the brand via Instagram/TikTok, needs strong trust signals (reviews, guarantees) before committing to a first purchase.
Returning Customer
Already purchased once, values a fast reorder experience, saved addresses, and recognition (loyalty, personalized recommendations).
5. Customer Journey Flow
The primary end-to-end customer flow this platform must support:
Landing  →  Browse Products  →  Product Details  →  Add to Cart  →  Checkout  →  Paystack  →  Order Confirmation  →  WhatsApp Confirmation  →  Shipping Updates  →  Delivery  →  Review  →  Repeat Purchase
Every stage in this flow maps directly to a documented screen/feature in Section 8, and every transition emits an analytics event so the funnel is fully measurable end to end (see Section 11, Analytics).
6. Free-Tier Cloud Architecture (No AWS)
AWS is explicitly excluded from this platform. The table below replaces every AWS component referenced in earlier planning with a free or near-free equivalent, chosen to minimize operating cost while the store is pre-revenue or early-revenue, with a clear upgrade path as traffic grows.

--- Table Start ---
Concern | Free-Tier Service | Why / Notes
Frontend hosting | Vercel (Hobby / free tier) | Next.js storefront + admin panel. Free tier covers generous bandwidth/build minutes for an early-stage store; automatic HTTPS, global CDN, preview deployments per PR.
Backend API hosting | Render (Free / Starter tier) or Railway (free credit tier) | NestJS API service. Free web service tier is sufficient for MVP traffic; documented upgrade path to a paid instance only when sustained load requires it.
Primary database | Neon (Postgres, free tier) — Supabase Postgres free tier as an alternative | Serverless Postgres with generous free storage/compute; branching supports safe staging/preview databases at no extra cost.
Caching / sessions | Upstash Redis (free tier) | Guest cart sessions, rate-limiting counters, short-lived stock holds during checkout.
Media storage & transforms | Cloudinary (free tier) | Product images, review photos, blog images; automatic responsive/format optimization (WebP/AVIF).
Transactional email | Resend (free tier) | Order confirmations, password resets, admin alerts.
WhatsApp messaging | Meta WhatsApp Cloud API (free conversation tier within Meta's free monthly conversation allowance) | Order confirmation, shipping updates, support click-to-chat.
Payments | Paystack | Card, bank transfer, and USSD payment channels; no separate PCI-DSS burden since Hairotic.ng never touches raw card data.
DNS / CDN / WAF | Cloudflare (Free plan) | DNS management, basic DDoS mitigation, and edge caching in front of Vercel/Render.
Error tracking | Sentry (free Developer tier) | Frontend and backend error monitoring with alerting.
Uptime monitoring | UptimeRobot (free tier) | External synthetic uptime checks feeding the 99.9% uptime KPI.
CI/CD | GitHub Actions (free minutes for public repos / limited free minutes for private) | Lint, test, build, and deploy pipelines on every pull request and merge to main.
Analytics | Vercel Analytics (free, basic) + custom event logging into Postgres | Web vitals/traffic from Vercel; ecommerce funnel analytics captured as first-party events (Section 11).
Secrets management | Platform-native encrypted environment variables (Vercel / Render project secrets) | No dedicated secrets-manager service required at this scale; documented rotation policy in the Security Suite.
--- Table End ---

Migration & Scaling Path
Every service above has a paid tier on the same platform (Vercel Pro, Render paid instance, Neon paid compute, Upstash pay-as-you-go, Cloudinary paid plan). This means scaling up is a configuration/billing change, not a re-architecture — no component requires migrating to AWS or another provider to grow.
7. Enterprise / Composable Commerce Architecture
The platform is organized as independently-owned modules rather than one monolithic block of features. Each module has a clear boundary, its own data ownership, and can evolve (or be swapped) without breaking the others. This is the same composable-commerce alignment referenced in prior planning, now mapped explicitly to screens and free-tier services.

--- Table Start ---
Module | Responsibility | Screens it powers
Catalog | Products, variants, collections, categories, search/filter, stock display. | Landing, Shop, Collections, Product Page
Customer | Registration, authentication, profile, addresses, wishlist. | Customer Dashboard, Wishlist, Admin Customer Management
Checkout | Cart, address capture, order creation, stock holds. | Cart, Checkout
Payments | Paystack integration, webhook verification, refunds. | Payment, Admin Orders
Notifications | Transactional email via Resend. | Order Confirmation, Contact
WhatsApp Messaging | Order confirmation, shipping updates, click-to-chat support. | Contact, Order Tracking
Analytics | Event capture, funnel reporting, admin dashboards. | Admin Analytics, Admin Dashboard
Administration | RBAC, admin auth, product/inventory/order management, audit logging. | Admin Login, Admin Dashboard, Product Management, Inventory, Orders, Customer Management
--- Table End ---

Module Boundaries — Implementation Note
At MVP scale, all modules can live in a single NestJS monorepo with clear internal module boundaries (NestJS modules mapping 1:1 to the table above) backed by one Neon Postgres database with clearly namespaced schemas/tables. This avoids the operational overhead of real microservices while preserving the ability to split any module out into its own service later without a rewrite.

2. Shop (Product Listing)

--- Table Start ---
Module | Catalog
Priority | Must Have (MVP)
--- Table End ---

Objective
Let customers browse the full catalog with filtering, sorting, and search so they can find the right product quickly.
Business Value
Primary discovery surface for customers who don't land on a specific Collection or product link.
Good filtering/sort reduces search abandonment and increases products-viewed-per-session.
Description
A paginated/infinite-scroll grid of all active products with filters (hair type, texture, length, price, color) and sort options (price, newest, popularity, rating).
User Story
As a shopper, I want to filter and sort products by what matters to me (length, texture, price) so I don't have to scroll through everything.
Acceptance Criteria
Filters can be combined (e.g., texture=curly AND length=18-20in) and reflected in the URL as query params for shareability.
Out-of-stock products are visibly marked but still appear (not silently hidden) unless admin disables them.
Search-within-shop returns results in under 500ms for catalogs up to 5,000 SKUs.
Grid supports at least 24 products per page with lazy-loaded images.
Edge Cases
Zero results after filtering: show a friendly empty state with a 'clear filters' action, never a blank grid.
Filter combination that matches nothing: suggest the nearest matching alternative (e.g., relax price range).
Product removed mid-session (admin deletes while user has tab open): handle gracefully on add-to-cart with a clear message.
Dependencies
Catalog service
Search index (e.g., Postgres full-text or Meilisearch free/self-hosted)
Inventory service (stock status)
Technical Notes
Use cursor-based pagination for performance at scale rather than offset pagination.
Search can start on Postgres `tsvector` full-text search (free, no extra infra) before graduating to a dedicated search engine if catalog grows beyond ~10k SKUs.
Analytics Events
shop_view  •  filter_applied  •  sort_applied  •  search_query  •  product_card_click  •  empty_search_result
Security Considerations
Sanitize and validate all filter/search query params server-side to prevent injection via crafted URLs.
Rate-limit search endpoint to prevent scraping/abuse.

Module | Checkout

Objective
Collect shipping/contact details and produce a confirmed, payable order with minimal friction.
Business Value
Directly gates the conversion KPI — every unnecessary field or step here costs completed orders.
Accurate address/contact capture reduces failed deliveries and support tickets.
Description
A guest-friendly, multi-step (or single-page) flow collecting contact info, delivery address, delivery method, and order review before handing off to Payment.
User Story
As a customer, I want to check out quickly, ideally without being forced to create an account first.
Acceptance Criteria

Address form supports Nigerian states/LGAs via a structured dropdown, not free text, to reduce delivery errors.

Phone number is validated and required (used later for WhatsApp order confirmation).

User abandons checkout mid-flow: recoverable via saved draft order tied to session/email for remarketing.
Delivery address outside serviceable areas: clearly communicate this before payment, not after.
Duplicate order submission (double-click on submit): idempotency key prevents creating two orders.
Dependencies
Customer service
Catalog/Inventory (final stock re-check)
Payments module (handoff)
Technical Notes
Generate the order in a 'pending_payment' state before redirecting to Paystack, using an idempotency key to prevent duplicate orders on retry/back-button.
Stock is soft-reserved (short TTL hold) at checkout start and confirmed on successful payment webhook.
Analytics Events
checkout_started  •  checkout_step_completed  •  address_entered  •  checkout_abandoned  •  order_created_pending
Security Considerations
All checkout traffic must be HTTPS only (enforced via Cloudflare/Vercel).
Validate and sanitize all address/contact fields server-side; never trust client-side validation alone.

8. Payment

--- Table Start ---
Module | Payments
Priority | Must Have (MVP)
--- Table End ---

Objective
Securely process payment via Paystack and reliably confirm order status without exposing sensitive payment data to the app's own servers.
Business Value
Payment reliability is a direct driver of completed-order rate and customer trust.
PCI-DSS scope stays minimal by never touching raw card data directly.
Description
Redirect/inline Paystack checkout for card, bank transfer, and USSD, followed by webhook-based server-side verification before marking an order as paid.
User Story
As a customer, I want to pay with my card, bank transfer, or USSD and get instant confirmation that my payment worked.
Acceptance Criteria
Order is only marked 'Paid' after Paystack webhook + server-side transaction verification, never on client-side redirect alone.
Failed/cancelled payments return the customer to checkout with cart intact and a clear retry option.
Successful payment triggers Order Confirmation screen, confirmation email, and WhatsApp message within seconds.
Supports Paystack's card, bank transfer, and USSD channels at minimum.
Edge Cases
Webhook delayed or fails to arrive: a reconciliation job polls Paystack's verify-transaction API as a fallback within a few minutes.
Customer closes the browser tab after paying but before redirect completes: webhook still confirms the order server-side independent of the client.
Duplicate webhook delivery (Paystack retries): handle idempotently using the transaction reference, don't double-fulfill.
Dependencies
Checkout (pending order)
Notifications module (email/WhatsApp)
Order Tracking
Technical Notes
Verify every webhook's signature against the Paystack secret before trusting the payload.
Store Paystack secret/public keys in the hosting provider's encrypted environment variables (Render/Vercel secrets), never in source control.
Reconciliation cron job (e.g., a scheduled Render job or GitHub Actions scheduled workflow) re-verifies any order stuck in 'pending_payment' for more than 10 minutes.
Analytics Events
payment_initiated  •  payment_success  •  payment_failed  •  payment_channel_selected
Security Considerations
Validate Paystack webhook signatures on every request; reject unsigned/invalid payloads.
No card data ever touches Hairotic.ng's own servers — Paystack hosted fields/redirect only.
Log payment events without storing sensitive payment details beyond transaction reference and status.

9. AI Development Guidelines
This section instructs any AI coding agent (or human engineer using AI-assisted tooling) on how to use this PRD alongside the technical specification to implement the application consistently.
What to Build
Exactly the 21 screens and 8 modules documented in Sections 6-8, in the priority order given (Must Have first).
Every acceptance criterion listed per screen — treat these as the literal definition of 'done' for that feature.
All free-tier cloud integrations listed in Section 6 — no AWS services, no paid-only infrastructure at MVP stage.
What NOT to Build
Native mobile apps (explicitly out of scope for MVP).
AI-driven product recommendations or AR try-on (out of scope for MVP; may be revisited post-MVP).
A multi-vendor marketplace model — Hairotic.ng is a single-seller storefront.
Subscription/recurring-billing products or a referral/affiliate program (post-MVP).
Any AWS service (S3, EC2, RDS, CloudFront, etc.) — use the Section 6 equivalents instead.
Custom-built secrets management, service mesh, or Kubernetes — unnecessary complexity at this scale.
Business Logic — Non-Negotiable Rules
An order is only 'Paid' after Paystack webhook signature verification succeeds — never on client-side redirect alone.
Stock is decremented atomically and only on confirmed payment, using row-level locking to prevent overselling.
Prices and totals are always recomputed server-side from the Catalog/Inventory source of truth; client-submitted prices are never trusted.
Guest checkout must always remain possible — account creation is optional, never mandatory, at any step.
Every order status transition is written to an immutable order_status_history record.
Feature Priorities
Build in this order: (1) all screens marked 'Must Have (MVP)', (2) 'Should Have (MVP)' screens, (3) 'Could Have (Post-MVP)' screens such as the Blog. Do not begin Should/Could items until all Must Have acceptance criteria pass.
Acceptance Criteria
Use the Acceptance Criteria list under each screen in Section 8 as the literal test checklist. A feature is not complete until every listed criterion is verifiably true, including the listed edge cases.
Success Metrics
Conversion rate > 2.5% (tracked via the funnel events defined in Section 11).
Platform uptime ≥ 99.9% (tracked via UptimeRobot).
Mobile page load < 2.5 seconds (tracked via Vercel Analytics / Web Vitals).
Repeat purchase rate > 20% and NPS > 50 (tracked via Customer Dashboard reorders and post-delivery review/survey data).
10. Non-Functional Requirements
Accessibility: WCAG 2.1 AA color contrast and keyboard navigability across the storefront, particularly given the deep-orange brand palette (verify contrast ratios against white backgrounds).
Performance: <2.5s page load on mobile 3G-equivalent; images served via Cloudinary responsive/auto-format.
Availability: ≥99.9% uptime, monitored externally via UptimeRobot with alerting to admin.
Scalability: stateless API instances behind Render's autoscaling (paid tier) as a documented upgrade path; database connection pooling (e.g., PgBouncer / Neon's built-in pooler) to handle burst traffic without an AWS-scale re-architecture.
Observability: structured logs, Sentry error tracking, and the Admin Analytics dashboard (Section 8, Screen 20) as the primary operational visibility tools.
11. Security
This PRD references the dedicated Hairotic.ng Security Suite for full implementation and operational standards rather than duplicating them here. Each feature in Section 8 states its security expectations (authentication, authorization, data protection); the Security Suite defines exactly how those expectations are implemented and audited.
Baseline Controls Assumed by Every Feature
HTTPS everywhere, enforced at the Cloudflare/Vercel edge.
JWT-based session auth with short-lived access tokens and httpOnly, Secure, SameSite=Strict cookies.
bcrypt password hashing (cost factor ≥12).
Role-Based Access Control (RBAC) enforced at the API layer for all admin endpoints.
Mandatory MFA for all admin/staff accounts.
Rate limiting on all public-facing write endpoints (auth, checkout, contact form, reviews) via Upstash Redis.
OWASP Top 10 mitigations (injection, broken auth, XSS, insecure deserialization, etc.) verified in CI before deploy.
Daily automated database backups (Neon's built-in point-in-time recovery on the free/paid tier) with a documented restore test cadence.
NDPR (Nigeria Data Protection Regulation) compliance for all customer PII handling, including a documented data-deletion process.
Refer to the Security Suite document for detailed threat models, incident response procedures, and audit logging schema.
